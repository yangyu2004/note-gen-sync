# 数据库

## 数据库三范式

* 列不可再分
* 行可以唯一区分
* 表的非主属性不能依赖于其他表的非主属性

## MySQL数据库引擎有哪些

* MYISAM:全表锁，拥有较高的执行速度，不支持外键，并发性能弱，占用空间相对较小，对事务完整性没有要求，以select、insert为主的应用基本上可以使用这个引擎
* Innodb:行级锁，提供了具有提交、回滚和崩溃回复能力的事务安全，支持自动增长列，支持外键约束，并发能力强，占用空间是MYISAM的2.5倍，处理效率相对会差一些。

## InnoDB与MYISAM的区别

* InnoDB支持事务，MYISAM不支持，对于InnoDB每一条SQL都默认封装成事务自动提交，这样会导致速度变慢，因此最好将多条SQL放在begin和commit之间，组成一个事务；
* InnoDB支持外键，而MYISAM不支持。对一个包含外键的InnoDB表转为MYISAM会失败
* InnoDB是聚簇索引，数据文件和索引是绑定在一起的，必须要有主键，通过主键索引效率很高。但是辅助索引需要两次查询，先查询到主键，然后通过主键查询到数据。因此主键不应该太大，会导致其他的索引也都会很大。而MYISAM是非聚簇索引，数据文件是分离的，索引保存的是数据文件的指针。主键索引和辅助索引是独立的。
* InnoDB不保存表的具体行数，执行select count(\*) from table时需要全表扫描。而MYISAM用一个变量保存了整个表的行数，执行上述语句时只需要督促该变量即可，速度很快
* InnoDB不支持全文索引，而MYISAM支持全文索引，查询效率上MYISAM更高

## 事务

什么是事务？多条SQL语句要么全部成功，要么全部失败。

事务的特性:ACID

原子性，一致性，隔离性，持久性

## SQL优化手段

* 查询语句中尽量不要使用select \*
* 尽量减少子查询，使用关联查询join替代
* 减少使用IN/NOT IN,用exists/not exists或者关联查询代替
* or的查询尽量使用union或者union all代替。
* 避免在where子句中使用!=或<>操作符，否则引擎将放弃使用索引转而对全表进行扫描
* 应该避免在where子句中对字段进行null值判断，否则将导致引擎放弃使用索引而全表扫描

## drop、delete与truncate的区别

SQL中的drop、delete与truncate均表示删除，但是三者有一些区别

delete和truncate只能删除表的数据而不删除表的结构

速度一般来说，drop> truncate >delete

delete的语句是DML，这个操作会放到RollBack segement中，事务提交之后才生效；如果有对应的trigger，执行的时候将被触发，truncate，drop是DDL，操作立刻生效，元数据不放到RollBack segement中，不能回滚，操作不触发trigger

## 什么是视图

视图是一个虚拟的表，具有和物理表相同的功能。可以对视图进行增改查。视图通常是有一个表或者多个表的行或者列的子集。对视图的修改不影响基本表。它使得我们获取数据更容易，相比多表查询

## 事务隔离级别有哪些？MySQL的默认隔离级别是？

SQL标准定义了四个隔离级别:

* 读取未提交
* 读取已提交 可以阻止脏读
* 可重复读 可以阻止脏读和不可重复读
* 可串行化 可以防止脏读、不可重复读、幻读

但是InnoDB在可重复读下使用的是Next-Key Lock算法，可以避免幻读的产生，达到了SQL标准的可串行化隔离级别。

## 大表如何优化？

* 1.限定数据的范围:禁止不带任何数据范围的查询语句
* 2.读/写分离 主库负责写，从库负责读
* 3.分库分表
  * 垂直分区 根据数据库里面的数据表的相关性进行拆分，即数据表的列部分，把一张列比较多的表拆分为多张表。
  * 优点：可以使得列数据变小，在查询的时候减少读取的Block数，减少I/O次数，此外可以简化表结构，易于维护
  * 缺点：主键冗余，需要管理冗余列，并会引起join操作，可以通过在应用层进行join来解决。此外，垂直分区会让十五变得更复杂
* 4.水平分区
  * 保持了数据库的结构不变，通过某种策略存储数据分片。这样每一片数据分散到不同的表或者数据库中，达到了分布式的目的。水平拆分可以支撑非常大的数据量。
  * 水平分表仅仅只是解决了单一表数据过大的问题，但由于表的数据还是在一个机器上，对于MySQL的并发能力并没有什么提升，所以水平拆分最好分库
  * 水平拆分能够支持非常大的数据量存储，应用端改造也少，但分片事务难以解决，跨界点join性能较差，逻辑复杂。尽量不要对数据进行分片，因为拆分会带来逻辑、部署、运维的各种复杂度。
  * 数据库分片的两种常见方案：
  * 客户端代理：分片逻辑在应用端，封装在jar包中，通过修改或者封装JDBC层来实现。
  * 中间件代理：在应用和数据中间加了一个戴利城。分片逻辑统一维护在中间件服务中。

* UUID:不适合作为主键，太长了而且无需不可读。比较适合用于生成唯一的名字的标示比如文件的名称

## MySQL中一条查询SQL是如何执行的？

```
select name from t_user where id = 1
```

1. 取得链接，使用使用到MySQL中的连接器
2. 查询缓存，key为SQL语句，value为查询结果，如果查到就直接返回。MySQL8.0在查询中已经将查询缓存删除。
3. 分析器，分为词法分析和语法分析，此阶段做一些SQL解析
4. 优化其，表中存在多个索引的时候决定用哪个索引；或者一个语句中存在多表关联join的时候，决定各个表的连接顺序
5. 执行器，。执行语句的时候还要判断是否具备此权限，没有权限就直接返回提示没有权限的错误；有权限则打开表，根据表的引擎定义，去使用这个引擎提供的接口，获取这个表的第一行，判断 id 是都等于 1。如果是，直接返回；如果不是继续调用引擎接口去下一行，重复相同的判断，直到取到这个表的最后一行，最后返回

## 索引有什么优缺点？

优点：提高数据的检索速度，降低数据库IO成本，索引之所以差的快是因为先将数据排好序。

缺点：占用存储空间，降低更新表的速度，因为不仅表要更新，表对应的索引也要更新

## MySQL所有类型有哪些？

主键索引：索引列中的值必须是唯一的，不允许有空值。

普通索引：MySQL中基本索引类型，没有什么限制，允许在定义索引的列中插入重复值和空值

唯一索引：索引列中的值必须是唯一的，但是允许为控制

全文索引：只能在文本类型CHAR,VARCHAR,TEXT类型字段上创建全文索引。字段长度比较大时，如果创建普通索引，在进行like模糊查询时效率比较低，这样可以创建全文索引。MyISAM和InnoDB中都可以使用全文索引。
